## 背景问题

以太坊里常见的钱包账户是 EOA（由私钥直接控制的外部账户）。它好用、兼容性也最好，但本身没有可编程的验证和执行逻辑。于是很多更顺手的体验只能依赖智能合约钱包或额外基础设施，普及会被兼容性、部署和迁移成本拖慢。

原文把问题集中在三类需求上：

1. **批量操作不够自然。** 例如在 DEX 中，用户通常要先发一笔 ERC-20 `approve`，再发一笔兑换交易。两笔交易意味着两次确认、两次等待，也可能在中间状态暴露风险。
2. **代付 gas 门槛高。** 应用想替新用户支付交易费，或用户想用 ERC-20 而非 ETH 支付服务费，传统 EOA 没有统一的账户级执行入口。
3. **私钥权限过粗。** EOA 私钥一旦签名，通常就是账户的完全权限。用户很难只授予某个会话密钥“每天最多花 1% 的某个代币、只能调用某个应用”这种受限能力。

我理解：EIP-7702 不是要淘汰 EOA，而是让既有 EOA 可以在不更换地址的前提下，接入一部分智能账户能力。

## 核心方案

EIP-7702 新增了交易类型 `0x04`。账户持有人签署一份 **authorization tuple（授权元组）**，其中包含链 ID、目标合约地址和 nonce 等信息。交易处理时，链会在该 EOA 的代码位置写入一个很短的 **delegation indicator（委托指示器）**：

```text
0xef0100 || 目标合约地址
```

它不是把整份钱包代码复制进用户账户，而是像留下一个“代码入口地址”。之后当外部调用这个 EOA 时，EVM 会取目标地址处的代码执行，但执行上下文仍是这个 EOA：余额、存储和地址都属于用户原来的账户。

这带来几个关键结果：

- 一个 EOA 可以调用经过审计的钱包实现，完成批量调用、会话密钥、限额、gas 代付等逻辑；
- 委托是**持久的**，不是只在当前交易结束前临时生效；用户也可以把目标地址设为零地址来清除委托，回到普通 EOA 状态；
- 授权元组会绑定 nonce，并可绑定当前链 ID（或用 `0` 表示不限定链），以控制重放范围；
- 授权列表会在交易主体执行前处理。即使交易主体随后 revert，已处理的委托也不会随之回滚；
- 协议只提供“委托到哪段代码”的底层能力，细粒度权限、签名重放保护、初始化和代付结算都要由被委托的钱包合约自己正确实现。

一个直观流程是：用户首次把 EOA 委托给某个钱包实现；之后在一次交互里，该钱包代码验证用户签名并依次执行“授权代币 → 调用 DEX → 清理或记录状态”。用户地址不变，但该次执行不再只能做单一 EOA 转账。

## 影响对象

| 对象 | 可能变化 |
| --- | --- |
| 普通用户 | 少确认一次或多次交易，可能获得免 gas、代币付费、会话权限和不换地址的钱包升级；同时必须理解“委托代码几乎等同于把账户控制权交给它”。 |
| 钱包与账户抽象开发者 | 可以让 EOA 复用智能账户实现，但需负责授权展示、初始化签名、nonce、存储隔离、撤销与迁移。 |
| DApp / DEX | 可把多步骤流程做成原子操作，降低新用户摩擦；但不能再把“这是 EOA”简单等同于“它没有代码或不会在一次调用中执行复杂逻辑”。 |
| Paymaster、Relayer、应用运营方 | 有了代付路径，但要处理用户让授权失效、先转走资产、导致代付方白烧 gas 的经济风险。 |
| 合约与安全审计人员 | 需要复查依赖 `tx.origin`、`msg.sender == tx.origin`、账户 code size/code hash 的旧假设。 |
| 客户端与节点实现者 | 要支持新交易类型、委托代码解析及交易池策略；原文建议，对带非零委托指示器的 EOA，交易池最多接受一笔待处理交易。 |

## 关键术语

| 术语 | 我的解释 |
| --- | --- |
| EOA | 由私钥控制、通常没有合约代码的外部账户。 |
| Set Code Transaction | EIP-7702 定义的 `0x04` 新交易类型，携带普通交易字段和授权列表。 |
| Authorization tuple | 用户对“哪个账户可委托到哪个地址、在什么 nonce/链范围内有效”的签名授权数据。 |
| Authority | 从授权元组签名恢复出的账户，也就是同意设置委托的 EOA。 |
| Delegation indicator | 写入 EOA 代码位的 `0xef0100 || address` 字节串，指向真正要执行的钱包代码。 |
| Delegate contract | 被指向的合约代码；它会在 EOA 的账户上下文中运行，因此对余额和存储的影响十分直接。 |
| Persistent delegation | 委托不会在交易结束时自动消失，除非另行更改或清除。 |
| Sponsorship | 第三方替用户垫付 gas，再按钱包规则结算的模式。 |
| Privilege de-escalation | 不交出主私钥，而签发能力更弱、更窄的子权限，例如限额会话密钥。 |
| `tx.origin` | 一笔交易最外层发起账户。EIP-7702 会改变部分合约对它与 `msg.sender` 关系的旧假设。 |

## 争议与风险

1. **“授权委托”很容易被误认为普通签名。** 原文明确认为，应用不应让用户直接对某段委托代码作授权；这段代码具有不受限的账户访问能力。若钱包 UI 只展示合约地址、不展示实现、升级规则和撤销方式，用户很难做出真正知情的确认。
2. **初始化可被抢跑。** 7702 不运行 initcode。若钱包在设置委托后再用一笔普通调用初始化存储，旁观者可能抢先调用初始化函数。规范要求初始化 calldata 由 EOA 密钥签名验证，但这把安全责任放到了钱包实现上。
3. **实现或升级失误会伤及原账户。** 委托代码使用 EOA 自己的存储。更换钱包实现时，存储槽碰撞、错误迁移或恶意升级都可能导致资产和权限问题。原文建议采用如 ERC-7201 一类的命名空间存储方案，但它不是协议层强制项。
4. **旧合约的安全假设会失效或变弱。** 原文指出，以 `tx.origin == msg.sender` 防原子 sandwich 或重入的写法可能受影响。即便这种写法本就不推荐，现实中仍有遗留合约，迁移期需要审计和兼容性测试。
5. **代付方会有经济攻击面。** 用户可以通过增加 nonce 使授权失效，或在执行前转空资产，让 Relayer 支付 gas 却拿不到约定报酬。保证金、风控和信誉系统会成为实际产品的必要组成。
6. **交易池与基础设施复杂度增加。** 被委托 EOA 可被其他调用影响余额或 nonce，使节点更难静态判断多笔 pending 交易是否仍有效。这并不是用户可见的功能，却是客户端必须承担的复杂度。

早期 Magicians 讨论中还出现过“委托代码是否应只在单笔交易临时存在”“是否应该允许 initcode”“授权签名是否必须覆盖 nonce 与 chain ID”等问题。最终规范选择了持久委托、地址指针和 nonce/可选链约束。我认为这说明 7702 的易用性不是无代价的：它有意增加一次更严肃的账户升级动作，来避免把钱包长期演化成随手签脚本的碎片化生态。

## 产品启发

1. **把账户升级设计成安全产品，而不是一个签名按钮。** 钱包应显示委托实现的名称、审计来源、权限范围、是否可升级、当前链、撤销入口和恢复方案；仅显示十六进制地址不够。
2. **优先做“可解释的批量操作”。** 对 Monad 上的 Swap、质押或 NFT 领取等多步流程，可以把步骤、资产变化、最大损失和失败条件预览给用户，再一次签名执行。原子性提升体验，但不应掩盖每一步的真实授权。
3. **会话密钥适合高频、低风险场景。** 例如游戏、打卡或自动复投提醒：限制合约白名单、代币种类、单笔/每日额度、有效期，并一键暂停。这样的权限模型比让 Agent 长期持有主私钥更可控。
4. **Gas 代付需要先有反滥用设计。** 不能只做“免 gas”营销；应配套 nonce 检查、报价有效期、预存保证金或信誉额度，避免 Relayer 被免费消耗。
5. **DApp 账户识别要从“EOA/合约二分法”升级。** 不要用 `extcodesize == 0` 或 `tx.origin` 作为唯一安全判断。把账户能力、签名验证方式和操作风险明确建模，才更适合 7702 与未来账户抽象共存的环境。

## 我的疑问

1. 钱包怎样用一个普通用户能理解的界面，解释“持久委托”和“本次交易授权”的区别，并避免用户把两者混为一谈？
2. 用户将 EOA 委托给可升级的钱包实现后，谁有升级权、升级延迟多久、紧急撤销如何做，才算足够可信？
3. 在 Monad 这类高吞吐 EVM 链上，7702 的批量调用会不会更容易让应用把复杂交易藏在一次确认中？钱包模拟和风险提示应如何适配？
4. 委托迁移时，历史存储如何可靠地兼容？如果旧实现没有命名空间存储或没有清理工具，用户的最小安全退出路径是什么？
5. Relayer 的保证金和信誉模型要如何设计，才能同时覆盖女巫攻击、资产波动和用户临时撤销，而不把代付成本又转回新用户？
