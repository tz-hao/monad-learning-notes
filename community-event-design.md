# 新人线上活动设计：AI Agent 上链，权限边界怎么设？

## 活动主题

```text
AI Agent 上链，权限边界怎么设？
```

一场 30–60 人的小型线上 Space / AMA。以 Monad 上的 `IntentGuard Agent Wallet` 学习原型为例，讨论 AI 如何协助生成交易建议，以及哪些权限必须被合约规则和用户确认约束。

## 活动目标

- 帮助新人建立正确预期：AI 可以解释、生成候选交易和策略，但不应直接持有私钥或替用户无条件签名。
- 用一个可理解的案例说明 `target`、函数 selector、交易金额上限和最终确认分别在保护什么。
- 让参与者在活动结束后能说清自己的第一个 AI x Web3 小实验应从哪条安全边界开始。

## 目标听众

- 刚开始接触 Monad、钱包或智能合约的 Builder Camp 学员。
- 想做 AI x Web3 Demo，但尚不理解签名、交易和合约权限关系的开发者。
- 对 Agent Wallet、session key、自动化交易感兴趣，但担心资产安全的产品或社区参与者。

建议人数控制在 30–60 人，便于提问、展示交易记录和围绕真实案例交流；不需要做成大规模宣发活动。

## 希望解决的核心问题

新人常把“AI 能生成交易”误解成“AI 应该自动执行交易”。这会忽略三个关键事实：

1. 私钥和最终签名权不能交给不受约束的模型。
2. 自然语言意图本身不够精确，必须被转成可检查的链上规则。
3. 高性能链不等于可以跳过权限控制；交易越快，错误执行后的风险也越需要提前限制。

活动用一个最小案例回答这个问题：用户允许 AI 调用 `CheckIn.checkIn()`，但禁止转出 MON、禁止调用非白名单合约。AI 可以提出调用建议，`IntentGuard` 则检查目标地址、函数 selector 和 `value` 是否符合策略，用户仍保留确认权。

## 为什么值得举办

AI Agent 和链上执行的结合正在成为 Builder 的常见方向，但“让 AI 帮忙”与“让 AI 控制钱包”之间有很大的安全差别。新人如果在第一步就理解权限边界，后续无论做交易助手、链上数据助手还是自动化工具，都能从可验证、可撤销、最小权限的设计开始。

这个主题也能把抽象的 Web3 概念落到真实学习证据上：本仓库已有 Monad 测试网 `CheckIn` 交互记录，交易 Hash 为 `0xd35c2b5174aa72e757ec6aef8d8a352c81391b0b453a507102f367896555d814`。活动不是空谈 Agent，而是从一次已经发生的合约调用出发，讨论如何给 AI 的后续调用加规则。

## 活动产出预期

参与者不需要现场完成完整产品。活动后能留下以下任一种轻量成果即可：

- 一条“允许什么 / 拒绝什么”的 Agent 策略 JSON。
- 一个 `checkIn()` 调用的候选交易与人工确认说明。
- 一份自己的 AI x Web3 权限边界清单。

这使活动服务于真实的下一步：为 Week 3 的 AI x Web3 项目先确定安全约束，再扩展功能和界面。
